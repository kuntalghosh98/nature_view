Nature View Project Context
===========================

Purpose
-------
Build a modern mobile-first website and admin-managed CMS for one environmental / eco-tourism / rural development project location.

The project is backed by an NGO/government land-use arrangement. The public website should not feel like an old NGO website. It should feel modern, trustworthy, nature-focused, and suitable for tourism, government credibility, public transparency, and community development storytelling.

The project may include:
- Nature viewpoints
- Eco resort / visitor attractions
- Gardens and plantations
- Farming initiatives
- Rural development programs
- Community employment and impact
- Events, awareness programs, and plantation drives

No visitor accounts are needed. No online booking or donation flow is planned for V1.


Primary Audiences
-----------------
1. Tourists / visitors
   Need to understand what they can see, how to visit, what facilities exist, and why the destination is worth visiting.

2. Government officials
   Need credibility, progress, compliance, impact, and transparent reporting.

3. Local community / volunteers
   Need information about participation, training, events, and employment/community work.

4. Partners / sponsors / media
   Need a credible story, visible impact, gallery, news, and contact channels.

Localization Strategy
---------------------
Static UI labels:
- Stored in frontend JSON translation files.
- Used for navbar labels, buttons, form labels, common static UI text.

Dynamic content:
- Stored in MongoDB with language fields.
- Example:
  title: {
    en: "Plantation Program",
    bn: "..."
  }

Fallback content:
- Backend will maintain JSON fallback files for public content.
- If DB content is missing or unpublished, backend returns fallback JSON.
- Frontend should receive the same response shape whether content comes from DB or fallback JSON.


Core Content Rules
------------------
- Every important public content area should be admin-managed.
- Content should support draft/publish.
- Public site should only show published DB content.
- If no published DB content exists, backend fallback JSON should be used.
- Media upload must be validated so UI does not break and website performance is protected.
- Admin panel should manage translations for dynamic content.

Avoid building all backend first or all frontend first. Each module should be completed end-to-end before moving to the next.

Next.js App Router Rules
------------------------

Metadata exports must only exist in Server Components.

Never use:

"use client";
export const metadata = ...

This causes Next.js build failures.

If a page requires:
- useState
- useEffect
- useReducer
- useRef
- useRouter
- usePathname
- useSearchParams
- Redux hooks

Then use the split-page pattern:

Example:

projects/
├── page.tsx
└── ProjectsPageClient.tsx

page.tsx:
- Server Component
- Owns metadata exports
- Imports and renders the client component

ProjectsPageClient.tsx:
- Contains "use client"
- Contains React hooks and browser-only logic

For public pages, prefer Server Components whenever possible.
Use client components only when browser interactivity is required.

Goals:
- Better SEO
- Better performance
- Better metadata support
- Smaller JS bundles
- Better mobile experience

Environment Setup
-----------------
Backend:
1. Copy backend/.env.example to backend/.env.
2. Set:
   - MONGODB_URI
   - JWT_SECRET
   - SUPER_ADMIN_EMAIL
   - SUPER_ADMIN_PASSWORD
3. Install dependencies in backend.
4. Run:
   npm run seed:admin
5. Start backend:
   npm run dev

Frontend:
1. Copy frontend/.env.example to frontend/.env.local.
2. Set:
   NEXT_PUBLIC_API_BASE_URL=http://localhost:5000/api
3. Install dependencies in frontend.
4. Start frontend:
   npm run dev

Admin URLs:
- /admin/login
- /admin/dashboard
- /admin/admins
- /admin/audit-logs
